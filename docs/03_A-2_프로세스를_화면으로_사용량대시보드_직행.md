# 03 A-2 프로세스를 화면으로 — 사용량대시보드 (직행)

> **1차 독자 선언문**: 이 산출물의 1차 독자는 B-1 단계 Claude Code(빌드 도구 AI). 플래너는 §1·§2·§3·§6·§7·§8만 직접 읽음. §4·§5 화면 상세 4-a/b/c/d는 빌드 AI가 읽고 구현. A-3 단계에서 시각 톤이 입혀지면 플래너가 결과로 검증.

## §1. A-1 요약 블록

**A-0 핵심 가설**
Primus Labs 개발자 10명이, Claude Code 세션 종료마다 자동 수집된 사용량 데이터를, 개인 대시보드 + 팀 랭킹으로 확인하면서 (1) 자기 워크플로우를 최적화하고 (2) 가벼운 경쟁을 즐길 것이다. **lower bar: 본인이 한 달에 한 번 보는 것만으로도 성공.**

**MVP 경계 3개**
- 제품: Claude Code 사용량 자동 수집 + 사내 웹 대시보드 (Codex/Gemini 제외)
- 타겟: Primus Labs 사내 개발자 10명 안팎
- 목표 행동: 주 1회 대시보드 방문 + 월 1회 자기 통계 보고 워크플로우 변경

**A-1 Phase 구조**
- P0. 온보딩 & 셋업 (한 번만, S1~S4)
- P1. 자동 수집 (UI 없음, S5~S6)
- P2. 조회 — 개인 + 팀 (S7~S9)

**스코프**: 1일 빌드 (8~12시간)
- AI 코드 작성 (Claude Code agent loop): 2~3h
- 본인 검토·디버깅: 2~3h
- Mac/Win 둘 다에서 init 테스트 (keytar OS 분기·hook merge 검증): 1~2h
- 친구 1명 초대해 onboarding 문제 잡기: 1~2h
- §7 운영 관제 노트 3개 항목 빌드: ~2h

**기술 스택**
- 백엔드: Next.js 14 API routes + Supabase Postgres (무료 티어)
- 프론트: Next.js 14 + Tailwind + Recharts + react-activity-calendar
- 인증: GitHub OAuth + 회사 이메일 도메인 화이트리스트
- 데이터 수집: ccusage를 npm 라이브러리로 import + Claude Code SessionEnd hook
- 비밀 저장: keytar (Mac Keychain / Win Credential Manager / Linux libsecret 자동 분기)
- 배포: Vercel + Supabase

## §4. 핵심 화면 상세

### 화면 #3 — 개인 대시보드 메인 ★

**4-a. Wireframe**

```
┌──────────────────────────────────────────┐
│ Primus Usage     [개인][팀랭킹][셋업]    │
│                       eugene ▾           │
├──────────────────────────────────────────┤
│ ⓘ 과거 데이터 수집 중 · 잠시 후 추가됩니다 ✕│ ← dismissable
├──────────────────────────────────────────┤
│ 오늘 12.3M tok · $4.20 · 3시간 활동       │
├──────────────────────────────────────────┤
│ [오늘][이번주][이번달][전체]              │
│ ┌─ 일별 토큰 (7일) ────────────────────┐ │
│ │     ▆       ▇                        │ │
│ │  ▅  █   ▄   █     ▃                  │ │
│ │  █  █   █   █     █   ▆              │ │
│ │ Mon Tue Wed Thu Fri Sat Sun          │ │
│ └──────────────────────────────────────┘ │
│ 모델별 (이번 주)                          │
│ Sonnet ████████████████ 67%  $32         │
│ Opus   ████████ 28%          $48         │
│ Haiku  █ 5%                  $1          │
│ 효율 지표 (팀 평균 비교)                   │
│ One-shot rate  78%  (팀 71%)             │
│ Cache hit      67%  (팀 67%)             │
│ 5h utilization 43%  (팀 52%)             │
│ 💡 제안 (3건 / 5건)         [더보기 →]   │
│ • Cache hit 낮음 [신뢰도:낮음]            │
│ • Opus 비중 높음                          │
│ • MCP 미사용                              │
└──────────────────────────────────────────┘
```

**4-b. UX 의도**
- 키 메트릭 한 줄 최상단 — lower bar 가설 직접 지원. 들어와서 1초 안에 "최근 어떻게 썼는지" 잡힘
- 모델별(양) → 효율(질) 순서로 본인 원칙 시각화
- 팀 평균 괄호 — 10명 규모에선 평균이 의미 있게 작동
- 제안 미리보기에서 신뢰도 라벨은 "낮음"만 텍스트 노출. 중·높음은 라벨 없음(시각 압박 줄임). 사용자가 "낮음" 표시된 것만 가중치 두게
- backfill 진행 배너 dismissable — 처음 한 번만 인지하면 됨

**4-c. 인터랙션**
- 기간 토글 → 모든 섹션 한 fetch로 갱신
- 차트 막대 hover → 그날의 토큰·비용 툴팁
- 효율 지표 행 클릭 → #4로 + 해당 메트릭 섹션 스크롤
- "더보기 →" 클릭 → #4
- 상단 탭 → #5 또는 #7
- backfill 배너 ✕ → 영구 dismiss
- 프로필 ▾ → 로그아웃

**4-d. 상태**
- 정상 (위 와이어프레임)
- **init 직후 첫 진입 + backfill 진행 중**: 차트는 init 이후 데이터로 즉시 표시, 상단 배너 노출. backfill 완료되면 차트 자동 갱신 (페이지 새로고침 불필요)
- **데이터 0건 (init 후 사용 0)**: "아직 Claude Code 세션이 없네요. 한 번 사용 후 종료해보세요" + 셋업 가이드 링크
- **마지막 수집 24h 초과 (hook 의심)**: 상단 노란 배너 "마지막 수집: 3일 전. 셋업 확인 →"
- 로딩: 차트 영역 skeleton

**4-b. UX 의도**
- Today's MVP는 합성 점수(`토큰 × one-shot rate`) 1명 부각. 매일 다른 사람 나올 가능성 ↑
- 한 줄 멘트는 룰 베이스 자동 생성 (LLM 호출 없음 — 비용·복잡도 0)
- 최다/최고 효율 동등 크기 병렬 — 한쪽 더 크면 그게 "정답"인 인상. "많이 vs 잘"의 동등성 시각화
- 10명 전체 노출 — 5위 이하 소외 방지
- 프로젝트별 하단 — 진짜 의미는 같은 프로젝트 안 비교 (사주톡 안에서 누가 잘 쓰는지)

**4-c. 인터랙션**
- 기간 토글 → 모든 카드 갱신 (Today's MVP는 "이 기간 MVP"로 라벨만 변경)
- Today's MVP 클릭 → #6
- 최다/최고 효율 행 클릭 → #6
- 프로젝트 드롭다운 → 다른 프로젝트로 전환
- 메트릭 토글 → 정렬 기준 변경 (토큰/one-shot/비용)

**4-d. 상태**
- 정상
- **팀 1명만 가입**: "아직 1명만 가입했어요. 다른 멤버 초대" + 초대 링크 복사 버튼
- **이 기간 활동 0** (예: 주말): "이 기간에 활동 없어요. 다른 기간 선택 ↑"
- **Today's MVP 동률**: 합성 점수 동률 시 알파벳 순 첫 사람 + 작은 글자 "+ seoyeon (동률)"

**라벨 규칙**:
- 낮음 (cache hit 룰, retry 룰): "[신뢰도:낮]" 명시
- 중·높음 (Opus 비중, 5h util, MCP 미사용): 라벨 없음

**검증 데이터 수집** (v2 분석용):
- Done 버튼 클릭 → DB에 `(user, suggestion_type, action_taken_at)` 로깅 → "본인이 진짜 행동했나" 측정
- 👍/👎 클릭 → DB에 `(user, suggestion_type, sentiment)` 로깅 → "이 추정이 맞다고 느꼈나" 측정
- v1 운영 1~2개월 후 데이터 보고 임계값 튜닝 + 라벨 점진 제거

의도: 평소엔 안 보지만 "마지막 수집 24h 초과 배너" 클릭하면 여기로. 가장 흔한 3개 액션만 노출.

## §6. [SO 결정 필요] — 모두 확정

- **SO1 Backfill**: 옵션 A (전체 backfill, 첫 인상 보호). 단 init 종료 조건에서 분리해 백그라운드로 진행 (§7 운영 관제 노트 참조)
- **SO2 공개 범위**: 옵션 A (모두 공개, 10명 규모. 단 본인 #4의 제안·5개 룰은 본인만)
- **SO3 랭킹 메트릭**: 옵션 C (최다/최고효율 두 카드 병렬 + Today's MVP 합성 점수)
- **SO4 Optimization**: 옵션 A (5개 룰 + 신뢰도 라벨 "낮음"만 + Done 버튼 + 피드백 👍/👎. 검증은 데이터 쌓고 v2)
- **SO5 One-shot 정의**: 옵션 A (edit/write 도구 호출이 첫 시도에 success로 끝난 비율)
- **SO6 누락 방지**: 옵션 A (SessionEnd hook만, ccusage가 jsonl 통째 파싱)

## §7. 운영 관제 노트 (B-1·B-2 빌드 시 반영)

A-2 마지막 라운드 적대적 압박에서 식별된 빌드 직전 위험 3개 + 완화책. 빌드 도구가 §4·§5 화면 외에 추가 구현해야 할 항목.

### 7-1. 신뢰도 라벨 검증 데이터 수집

**위험**: "[신뢰도:낮음]" 라벨이 박힌 제안은 사용자가 무시 → Done 클릭 0 → 검증 데이터도 안 쌓임

**완화책 (구현됨, §4 #4)**:
- 신뢰도 라벨은 "낮음"만 텍스트 노출. 중·높음은 라벨 자체 없음
- 모든 제안 옆 "이 추정 어때요?" 👍/👎 미니 버튼 (Done과 별개)
- DB 스키마에 `suggestion_feedback` 테이블 — `(user_id, suggestion_type, sentiment, created_at)`

### 7-2. ccusage 의존성 깨짐 모니터링

**위험**: Anthropic이 jsonl 스키마 변경 → ccusage 업데이트까지 며칠 → 그 사이 데이터 수집 중단 → 본인이 매일 #7 셋업 상태 확인해야 함

**완화책**:
- 일일 cron (Vercel Cron 또는 Supabase Edge Function)으로 "마지막 24h 수집 0건 멤버" 쿼리
- 0건 멤버 1명 이상 발생 시 Slack 사내 채널 webhook으로 알림
- 알림 내용: "@eugene 사용량 수집 24h 끊김: jiwoo, minho. ccusage 또는 hook 점검 필요"
- 빌드 시간: ~1h (Slack webhook 등록 + cron 함수 + 쿼리)

### 7-3. Backfill 첫 인상 보호

**위험**: 6개월치 jsonl이 수십 MB → ccusage 첫 init이 30초~2분 → 셋업 가이드 #2가 멈춰 있는 것처럼 보여 사용자가 닫음

**완화책 (구현됨, §4 #2·#3)**:
- npx init의 종료 조건에서 backfill 제외. Node 확인·keytar·hook 등록 3개만 끝나면 #3로 자동 이동
- backfill은 init이 종료된 후 백그라운드 자식 프로세스로 분리 실행
- #3 진입 시 상단에 dismissable 배너 "과거 데이터 수집 중 · 잠시 후 추가됩니다"
- backfill 완료 시 #3 차트가 자동 갱신 (Supabase realtime 또는 5초 polling)

## §8. Kill / Go / Hold 판정

**Go**

근거 3줄:
- 핵심 가설 lower bar("본인이 한 달에 한 번 보면 성공")가 명확. 빌드 후 검증 가능
- 1일 스코프 + Vercel/Supabase 무료 티어 = 운영 부담 0. 사주 1일 거리두기 안에 들어감
- 모든 [SO 결정] 항목 확정. 운영 관제 위험 3개에 완화책 박힘. 빌드 단계로 진입 가능

## 적대적 압박 — Kill할 이유 (참고용 누적)

### 1차 라운드 (턴 1, 빌드 가치 의문)
1. 행동 변화 가설이 검증 안 됨 → **Kill 1로 lower bar 재정의 ("본인이 월 1회 보면 성공")**
2. 사내 10명 규모에서 랭킹 동력이 빠르게 식음 → **매니지먼트 채널 별도 존재로 보완**
3. 본업(사주톡) 기회비용 → **사주 품질 검증 막힌 상태, 1일 거리두기로 대체 활동**

### 2차 라운드 (턴 4, 빌드 직전 위험)
1. 신뢰도 라벨이 "다 무시" 행동 유발 → **§7-1 완화책**
2. ccusage 의존성 깨짐 → **§7-2 완화책**
3. Backfill 첫 인상 망침 → **§7-3 완화책**

3개 다 1일 스코프 안에 완화책 들어감. Kill 트리거 아님.

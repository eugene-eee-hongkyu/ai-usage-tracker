## §3. 기술 리스크 사전 조사 + Hold 플래그

**리스크 1: ccusage 호출 방식 — 해소**
- 결정: child_process spawn으로 `npx ccusage --json --since YYYY-MM-DD` 실행해 stdout 파싱
- 버전 pin 없음 (매번 최신) — pin 두면 우리가 정기 업데이트 검증 책임 생김 → 본인 원칙 위반
- ccusage가 backward compatible JSON output 유지 약속 신뢰. 깨지면 #3 대시보드 상단 노란 배너("마지막 수집: N일 전")로 즉시 발견
- 라이브러리 의존 0

**리스크 2: Claude Code SessionEnd hook 발화 검증 — Hold 플래그**
- 공식 문서는 SessionEnd hook 명세 있으나 Windows 환경에서 cmd.exe·PowerShell 분기 동작 사례 부족
- **Hold 플래그**: 친구 1명(본인 외 OS)에서 init 후 세션 종료 → submit.mjs 실제 호출되는지 검증
- §10 시나리오 2에 검증 단계 명시. PASS면 B-2 진입. FAIL이면 cron fallback 추가 (별도 결정)

**리스크 3: keytar 네이티브 빌드 — 해소**
- prebuilt 바이너리 제공. Node 18+에서 자동
- 일부 Linux는 `libsecret-1-dev` 사전 설치 필요. init script에서 OS 감지 후 안내

**리스크 4: Auth.js stable 자동 선택 — 해소**
- Claude Code가 빌드 시점에 `npm view next-auth dist-tags` 또는 `npm view @auth/core dist-tags`로 stable 확인
- v5가 stable이면 v5 설치, 아니면 v4 설치
- 도메인 화이트리스트 callback hook은 v4·v5 둘 다 표준 패턴

## §10. 검증 방법

### 단계 1. 시나리오 PASS

```
[검증 1] 본인 init + 데이터 수집
  실행: 수동
  PASS 조건:
  - npx github:primus-labs/usage-tracker init 실행 → OAuth 자동 열림
  - "@primuslabs.gg" 이메일 통과, 다른 도메인 reject
  - Node 확인·keytar·hook 등록 3개 완료 후 web/setup이 자동 #3 이동
  - 상단 "과거 데이터 수집 중" 배너 노출
  - 1~3분 후 차트가 7일치로 채워짐 (백그라운드 backfill 완료)
  - Claude Code 새 세션 1회 실행 후 종료 → 5분 안에 sessions 테이블에 row 추가

[검증 2] 친구 1명 init (다른 OS) — Hold 플래그 §3 리스크 2 해소
  실행: 수동
  PASS 조건:
  - 본인 Mac에서 검증 1 통과 후, 친구 Windows에서 동일 init 실행
  - 친구 OS에서 keytar 정상 저장
  - 친구 ~/.claude/settings.json에 hook merge (기존 hook 보존)
  - 친구 새 세션 1회 → submit.mjs 실제 발화 (Windows에서 node 실행)
  - 팀 랭킹 #5에 두 사람 노출
```

## 적대적 압박 — Kill할 이유 세 가지 (빌드 직전 마지막)

**1. ccusage가 어느 날 호환 깰 위험.** 버전 pin 없이 매번 최신 받는 구조라 ccusage가 breaking change 내면 그날부터 데이터 0. 사용자는 #3 배너로 발견하지만 그동안 며칠치 잃음. **완화책 (이미 반영됨)**: 배너로 즉시 발견 가능. ccusage 깨지면 그때 옵션 결정 (다른 도구·자체 파서 등). 본인이 매일 cron 알림 거부했으므로 발견 늦을 수 있다는 것 인지하고 있음.

**2. Auth.js stable 자동 선택이 빌드 중간에 어느 버전 설치할지 결정 → 두 버전 사이 API 차이로 빌드 산출물 비결정적.** Mac에서 v5로 빌드, Win에서 같은 시점에 v4로 빌드되면 문제 (npm dist-tags 시간차). **완화책**: package.json에 한 번 설치되면 lockfile로 고정. 본인이 첫 빌드 후 git commit 하면 이후엔 모든 머신에서 같은 버전. 단, 첫 빌드 결과만 비결정적이라는 점 인지.

**3. 디렉토리 분리 (workspace) 셋업이 1일 빌드 시간 빠듯하게 만듦.** 본인이 "표준·확장성"으로 이걸 선택했으나, 빌드 끝에 시간 부족으로 화면 #6(멤버 프로필) 또는 5개 룰 중 일부를 v2로 미뤄야 할 수 있음. **완화책**: §9.5에 12h 상한 박혀있음. 12h 도달 시 Claude Code가 정지. 그 시점에 본인이 미룰 항목 결정. 우선순위 — 화면 #6 > 룰 5개 중 retry 평균 > 5h util (이 둘은 정의 모호 + 검증 어려움).
